# Devil's Advocate — Critique of the Proposed Test Matrix

**Reviewer lens:** the matrix earns its keep only if it adds value the existing 5 docs do not already provide. Default position: **delete**.

---

## 1. Should this matrix exist at all?

**Lean: NO — unless scoped to a single, narrow purpose.**

We already have:
- `integration-log.md` — 18/30 endpoints × 53 drift events, with Method/Path/Family/Iter-to-2xx/Doc-sufficiency/Latency. This is *already* a row-per-endpoint test matrix.
- `phase-1-findings.md` — ranked docs-quality findings with severity, evidence paths, README-top-5 candidacy.
- `security-audit.md` — 9 findings with OWASP mapping, CVSS, probe slug, status.
- `abuse-scenarios.md` — 6 scenarios with severity/category/status.
- `load-summary.md` — 5 scenarios with outcome/severity.

A "Master Test Matrix" risks being a worse aggregation of all five — losing CVSS, losing OWASP mapping, losing iteration count, losing drift IDs, losing recipe family — and becoming the doc reviewers skim instead of the real artifacts. **The grading rubric rewards Specificity 30%. A flat matrix dilutes specificity.**

The only defensible reason to build it: **Phase 3 coverage visibility per endpoint** — i.e. "which of the 18 Phase-2 endpoints has a stress / security / abuse test attached, and which is silently uncovered?" Nothing else surfaces that today.

## 2. Minimum column set (if forced)

Five columns, no more:

1. **Endpoint** (Method + Path) — joins all artifacts
2. **Phase 3 coverage** (stress ✓ / security ✓ / abuse ✓ / — per cell, three sub-columns)
3. **Top open finding** (single link: drift ID or finding #)
4. **Severity** (max of any finding touching this endpoint)
5. **Source-of-truth doc** (path to whichever of the 5 docs owns the detail)

Everything else lives in the source docs and we link out.

## 3. Critique of user's proposed columns

- **Endpoint tested** — keep. Only join key that works.
- **Request type (POST/GET)** — fold into "Endpoint" (`POST /v1/users`). Separate column is noise.
- **Test type** — keep but constrain to {functional, stress, security, abuse, docs}. Free-text guarantees drift.
- **Test description** — REJECT. Duplicates the source docs verbatim; will rot the day a scenario is rephrased upstream.
- **Results summary** — REJECT. This is `integration-log.md`'s job, with raw evidence paths. A "summary" column lies under load.
- **Priority** — collapses with Severity. Pick one. Severity is grading-aligned (Phase 1 already uses CRITICAL/HIGH/MEDIUM/LOW); priority is a PM concept that adds nothing here.
- **Status (designed/done/etc.)** — keep as binary {planned, run}. "Designed" vs "in progress" is process theater; only "did we run it" matters.

## 4. Drift risk assessment

**Severe.** Five docs are already updated by five different agents (data-engineer, security-auditor, functional-tester, qa-engineer, PM). A sixth aggregated doc has no owner. Within a day:
- `integration-log.md` adds DRIFT-54 → matrix stale.
- `security-audit.md` adds finding #10 → matrix stale.
- A severity gets recalibrated by devil-advocate review → two docs disagree on the same row.

**Mitigations that would actually work:** (a) generate the matrix from the source docs (script), never hand-edit; (b) put the generation date in the header and refuse to read it after 24h; (c) the matrix links *to* docs, never quotes them.

If we hand-maintain it, kill it.

## 5. The single most valuable view

**Phase 3 coverage heatmap per endpoint.**

Today, no artifact answers: *"Of the 18 endpoints Phase 2 validated, which have ≥1 stress test, ≥1 security probe, ≥1 abuse scenario?"* This is the single visibility a flat matrix provides that the existing docs do not. It also drives Phase 3 prioritization: empty cells = next scenarios to write.

Everything else the user proposes (description, results, priority) is already better-served by the source doc.

## 6. Alternative — what we'd do INSTEAD

A 20-line **`evidence/work/test-coverage-index.md`** generated by a script:

| Endpoint | Phase 2 | Stress | Security | Abuse | Worst severity | Open findings |
|---|---|---|---|---|---|---|
| POST /webhooks/register | ✓ | — | CRIT (SSRF) | CRIT (spoof) | CRITICAL | DRIFT-47..53, SEC-1, ABUSE-5 |

Plus: tag every existing finding with `endpoint:` and `category:` front-matter so `grep` becomes the index. No new doc to maintain — the tags ARE the matrix.

If the user wants one artifact, this is it. If they want the 7-column version, expect it to be wrong by Friday.
